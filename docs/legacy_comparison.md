# 旧エンジン (WGCS) と新エンジン (Unified GameCore) の機能比較と統合方針

統合にあたって、オリジナルの `WebGameCoreSystem` (以下 WGCS) の機能のうち、一部を新エンジンのコアには含めない、あるいは別の仕組みで置き換える判断をしました。その根拠を説明します。

---

## 1. 統合・移行された主要機能

| カテゴリ | WGCS の機能 | 新エンジンでの対応状況 |
| :--- | :--- | :--- |
| **描画** | `DisplayControl` (Layers) | ✅ `DisplayManager` として統合。 |
| **描画** | `spPut` (高度なスプライト描画) | ✅ `Layer.spPut` として完全移植。 |
| **スプライト** | `GameSpriteControl` | ✅ `SpriteSystem` として統合。 |
| **サウンド** | `Beepcore` | ✅ シンセサイザー機能を完全移植。 |
| **入力** | Keyboard, Mouse, Gamepad | ✅ `InputMapper` による論理抽象化として再設計。 |
| **アセット** | `GameAssetManager` | ✅ `AssetManager` として非同期ロード対応。 |

---

## 2. 意図的に統合しなかった機能とその根拠

新エンジンでは「**最小限のコア ＋ 拡張可能なコンポーネント**」という設計思想を採用しています。

### ① `GameTask` / `GameTaskControl` (タスク管理システム)
- **内容**: `init`, `pre`, `step`, `draw`, `post` を持つ「タスク」を配列で管理し、`signal` で通信する仕組み。
- **不採用の根拠**:
    - **新方式**: `SceneNavigator` (高レベル状態) ＋ `EntityManager` (ECS的な実体) ＋ `EventBus` (イベント駆動) の組み合わせで完全に代替可能です。
    - **理由**: 従来の `signal` による直接通信よりも、`EventBus` の方がオブジェクト間の結合度が低く、現代的な設計に適しています。また、`Entity` と `Behavior` の組み合わせで、より柔軟なロジック構築が可能です。

### ② `fontPrintControl` (ビットマップフォント描画)
- **内容**: SHIFT-JIS などのビットマップフォント画像から文字を切り出して描画する機能。
- **不採用の根拠**:
    - **新方式**: Canvas API 標準の `fillText` を使用。
    - **理由**: 現代のブラウザ Canvas は高いパフォーマンスで任意のフォント・UTF-8 文字列を描画可能です。ビットマップフォントは特定のアートスタイル（レトロ等）に依存するため、エンジンコアに含めるのではなく、必要に応じて `Behavior` 等の拡張モジュールとして追加するのが適切です。

### ③ `textConsole` (コンソールエミュレータ)
- **内容**: ncurses のように文字列のバッファを持ち、スクロールやカーソル制御を行うターミナル風の描画機能。
- **不採用の根拠**:
    - **理由**: これは非常に特殊な用途（ローグライクやデバッグ用コンソールなど）に特化した機能であり、汎用ゲームエンジンの「コア」としては肥大化の原因となります。新エンジンの `EntityManager` をベースに「コンソール表示用の Entity」を自作する方が、描画レイヤーの制約を受けず柔軟です。

### ④ `inputVirtualPad` / `inputTouchPad` (モバイル用 UI)
- **内容**: 画面上にボタンを表示し、タッチ入力を擬似的にキー入力に変換する機能。
- **不採用の根拠**:
    - **理由**: バーチャルパッドの見た目（テクスチャ）や配置はゲームデザインに強く依存します。新エンジンでは `InputMapper` による抽象化レイヤーを提供しているため、ゲーム側の `Scene` 上に UI 部品として実装する方が、各プロジェクトの要件に合わせやすくなります。

---

## 3. その他の細かな変更

- **`blink()` ユーティリティ**: 1.5秒間隔で点滅する真偽値を返す機能。特定のリズムに依存するためコアからは除外し、ゲームロジック側で `Math.sin` 等を用いて実装することを推奨します。
- **`benchmark` 機能**: フレームレートや負荷をミリ秒単位で計測する専用クラス。新エンジンでは `GameCore.status` に基本的な `frame` や `deltaTime` を持たせており、詳細なプロファイリングはブラウザの `Performance` タブ（DevTools）の利用を前提としています。

---

## 結論：新エンジンの方向性
新エンジンは、WGCS が持っていた「便利なツールセット」としての側面を整理し、**「ゲームの骨格」としての堅牢さと拡張性**を優先しました。

統合しなかった機能は「不要」になったのではなく、「**コアエンジンを軽量に保ち、プロジェクト固有の拡張（Behavior や Scene）として実装すべきもの**」として分離した形になります。これにより、将来的なメンテナンス性が向上し、多様なゲームジャンルへの対応が容易になっています。
